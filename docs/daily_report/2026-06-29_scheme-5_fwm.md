## 현재 정의와 구조

### 1. Seeded FWM / squeezing 모드

- 이 경로는 여전히 `85Rb D1 double-Lambda` seeded FWM을 유지한다. `compute()`는 `mode != Biphoton`일 때 `compute_spectrum()`으로 내려가고, 기본값도 Sim et al. 동작점에 맞춰져 있다 (`gabes/schemes/fwm.py:1860-1884`, `gabes/schemes/fwm.py:1726-1739`).
- seeded 계산은 실제 `85Rb D1` hyperfine strength와 branching을 쓰고, branch summation을 금지한다. 즉 서로 다른 Raman branch를 억지로 한 susceptibility에 합치지 않게 막아 둔 점이 물리적으로 맞다 (`tests/test_regression.py:52-83`, `gabes/schemes/fwm.py:1290-1299`).
- `line_strength`는 더 이상 “주 coupling 전체”가 아니라, first-principles normalization 위에 얹는 residual calibration으로 설명된다. 코드도 `coupling_norm`과 `effective_line_strength`를 분리해 반환한다 (`gabes/schemes/fwm.py:1325-1333`, `gabes/schemes/fwm.py:1488-1490`, `gabes/schemes/fwm.py:1620-1632`).
- high-density runaway는 bare gain을 그대로 보여 주지 않고, `pump_depletion_saturation()`으로 energy-conservation cap을 씌운다. 절대 gain이 비정상적으로 폭주하는 것을 실험가 관점에서 해석 가능하게 만든다 (`gabes/observables.py:87-115`, `gabes/schemes/fwm.py:1450-1459`).

### 2. Biphoton / generic SFWM 모드

- 같은 스킴 안에서 `mode = Biphoton`으로 바뀌면 `GenericFWMSolver(topology)`가 `87Rb telecom`, `133Cs BTW`, `generic diamond` topology를 푼다 (`gabes/schemes/fwm.py:569-590`, `gabes/schemes/fwm.py:1572-1719`, `gabes/schemes/fwm.py:1860-1865`).
- predictive 모드는 단순 reference waveform 주입이 아니라, Doppler-averaged amplitude, velocity-class coherent sum, `Omega_c^2` AT term, natural-linewidth decay, optional longitudinal reshaping까지 포함한다 (`gabes/schemes/fwm.py:633-725`, `gabes/schemes/fwm.py:778-820`).
- 특히 현재 HEAD에서는 predictive telecom width가 coarse velocity step aliasing에 끌려다니지 않도록 `_predictive_velocity_step()`과 `_converged_predictive_waveform()`이 자동 refinement를 수행한다 (`gabes/schemes/fwm.py:742-820`).
- 다만 absolute pair rate는 여전히 reference-anchored scaling이다. 코드가 아예 `pair_rate_cps_per_mw` anchor, density scaling, vector phase-matching weight를 곱하는 구조를 명시한다 (`gabes/schemes/fwm.py:870-880`).
- biphoton readout도 정직하다. validation table은 calibrated 항목을 `by construction`, predictive 항목을 `predicted (approx)`로 표기해 “맞춘 값”과 “물리적으로 나온 값”을 구분한다 (`gabes/schemes/fwm.py:2193-2245`).

## 직접 실행한 결과

실행:

- `python tests/test_regression.py`
- `python tests/test_fwm_generic.py`

결과:

- 두 테스트 모두 통과했다.
- seeded regression은 frozen baseline 유지, branch summation 금지, hyperfine strength/branching 일치를 확인한다 (`tests/test_regression.py:42-98`).
- generic biphoton 테스트는 vector phase matching, detector background/CAR, timing jitter broadening, predictive width ordering, 2D phase map, UI path까지 함께 확인한다 (`tests/test_fwm_generic.py:65-228`, `tests/test_fwm_generic.py:360-437`).

추가로 오늘 직접 계산한 대표 수치는 다음과 같다.

- seeded 기본값에서 `max G_s ~= 6.99`, `min S_dB ~= -8.44 dB`였다.
- seeded에서 `temp_c = 110 -> 121 -> 130 C`로 올리면 `max G_s ~= 1.65 -> 6.99 -> 105.87`로 급격히 커졌다. README가 경고한 “고밀도 exponential sensitivity”와 같은 방향이다 (`README.md:161-166`, `gabes/observables.py:87-115`).
- `87Rb telecom` biphoton은 calibrated에서 `FWHM ~= 0.534 ns`, predictive에서 `FWHM ~= 0.497 ns`가 나왔고, predictive 경로는 `velocity_step_used ~= 0.541 m/s`, `velocity_converged = True`였다 (`gabes/schemes/fwm.py:990-993`, `gabes/schemes/fwm.py:2231-2237`).
- `Cs BTW` predictive에서는 `852-917 nm` 채널이 `FWHM ~= 0.582 ns`, `852-795 nm` 채널이 `FWHM ~= 2.656 ns`로 더 넓었다. 현재 테스트가 요구하는 ordering과 일치한다 (`tests/test_fwm_generic.py:200-213`).
- timing은 warmed run 기준으로 seeded `compute() ~= 824 ms`, `observables() ~= 147 ms`, predictive telecom `compute() ~= 79-93 ms`, `observables() ~= 420 ms`였다. 즉 biphoton predictive의 physics 보강은 계산 부하를 늘리지만, 지금 체감 병목은 여전히 figure 생성 쪽이 더 크다 (`gabes/schemes/fwm.py:1892-2251`).

## 이전 개선안 추적과 계산 부하 평가

이번 스킴에서 직접 이어지는 기존 개선안은 저장소 안에 이미 남아 있다.

### 1. `biphoton-predictive-model`

- 위치: `docs/checklist.json:14-18`
- 상태: `in_progress`
- 이전 2026-06-24 보고서에서는 predictive telecom width가 아직 reference급이 아니고 aliasing 문제가 남아 있다고 봤다. 그런데 현재 checklist 설명과 HEAD 코드를 보면, auto-refined velocity grid가 이미 들어와 있다 (`docs/checklist.json:16-17`, `gabes/schemes/fwm.py:742-820`).
- 실제로 오늘 재측정한 predictive telecom width가 `~0.50 ns`로 내려와 reference `0.56(4) ns`에 가까워졌다. 6월 24일 보고서의 `~4.85 ns` 상태에서 실질적으로 크게 개선된 셈이다 (`docs/daily_report/2026-06-24_scheme-5_fwm.md`, `gabes/schemes/fwm.py:2231-2237`).
- 계산 부하는 분명 조금 생긴다. calibrated telecom compute가 `~15 ms`였고 predictive는 `~79-93 ms`였다. 하지만 solver 차원 증가가 아니라 velocity-grid refinement 정도이므로, 인터랙티브 사용을 깨뜨릴 정도는 아니다. 게다가 현재 총 체감 비용은 plotting이 더 크다.
- 결론적으로 이 개선안은 “부하는 조금 늘지만, 물리를 따라가는 정도는 크게 좋아지는” 성공적인 trade-off다.

### 2. `fwm-per-source-biphoton-calibration`

- 위치: `docs/checklist.json:42-45`
- 이 항목은 아직 deferred다.
- 현재 predictive가 waveform ordering과 수렴성은 확보했지만, absolute ns width와 topology별 effective linewidth/dephasing은 여전히 lumped approximation에 남아 있다. 코드도 이를 help/validation table에서 계속 인정한다 (`gabes/schemes/fwm.py:2072-2094`, `gabes/schemes/fwm.py:2231-2237`).
- 이 개선은 topology별 scalar anchor를 다루는 성격이라, 계산 부하는 거의 늘리지 않으면서 실험 레퍼런스 가치만 올리기 좋은 편이다.

### 3. `fwm-line-strength-factorization`

- 위치: `docs/checklist.json:49-52`
- seeded 쪽 `line_strength` residual을 overlap, polarization purity, effective participating Zeeman fraction 같은 lab-facing residual로 쪼개자는 제안이다.
- 현재 코드도 `line_strength`가 “실험적으로 직접 조절하는 진짜 fundamental knob는 아니다”라고 정직하게 적고 있다 (`gabes/schemes/fwm.py:1620-1632`).
- 이 분해 역시 solver dimension을 늘리지 않는 현상론적 정리라 연산 부하는 사실상 거의 없다. 대신 실험가가 “왜 0.74가 필요한가”를 훨씬 납득하기 쉬워진다.

## 실험물리학 연구자 관점의 평가

### Seeded FWM는 얼마나 실험 레퍼런스로 쓸 만한가

- 오늘 기준 seeded 모드는 **warm-vapor 85Rb D1 twin-beam FWM의 동작점 탐색과 감도 스캔용 semi-quantitative reference**로 충분히 쓸 만하다.
- 이유는 명확하다. 실제 hyperfine strength와 branching을 쓰고 (`tests/test_regression.py:65-83`), wrong branch summation을 금지하고 (`gabes/schemes/fwm.py:1290-1299`), paper anchor와 detector loss를 함께 얹어서 gain과 squeezing을 한 readout 안에서 읽게 해 주기 때문이다 (`gabes/schemes/fwm.py:1895-1989`).
- 특히 temperature를 올렸을 때 gain이 급격히 좋아지다가 saturation cap 논리가 필요해지는 모습은, 실험실에서 “왜 수치가 너무 좋게 나오지?”를 빨리 감지하게 해 준다. 이건 실험 planning 도구로 중요하다.
- 다만 **최종 absolute source metrology reference**로 쓰기에는 아직 조심해야 한다. full 24-level Zeeman/Floquet main solve는 아니고 (`README.md:208-218`, `tests/test_fwm_generic.py:360-365`), `line_strength` residual도 아직 하나의 lumped 보정에 남아 있기 때문이다.

### Biphoton 모드는 얼마나 쓸 만한가

- biphoton 모드는 **geometry, phase matching, filter bandwidth, detector background, timing jitter까지 포함한 source-design reference**로 상당히 유용하다.
- 특히 vector phase matching을 longitudinal scalar 하나로 뭉개지 않고 transverse mismatch까지 suppression시키는 점이 실제 정렬 실험에 바로 도움이 된다 (`gabes/schemes/fwm.py:931-977`, `tests/test_fwm_generic.py:65-120`).
- predictive 경로가 이제 telecom width 수렴을 어느 정도 잡았다는 점도 크다. 예전보다 “qualitative toy”에서 한 단계 올라왔다.
- 그러나 absolute pair rate는 여전히 anchor 기반이고 (`gabes/schemes/fwm.py:870-880`), per-source linewidth/dephasing calibration이 아직 남아 있다 (`docs/checklist.json:42-45`). 따라서 **실험 경향성과 geometry sensitivity를 보는 reference**로는 좋지만, 절대 발생률/절대 파형폭의 최종 기준기로 쓰기에는 아직 이르다.

## 순수 코딩 관점의 최적화 포인트

물리를 건드리지 않고 속도를 올릴 수 있는 포인트도 보인다.

### 1. figureless / headless observables 경로

- 현재 seeded와 biphoton 모두 `observables()`가 항상 Matplotlib figure를 만든다 (`gabes/schemes/fwm.py:1892-2251`).
- 오늘 timing에서도 predictive telecom은 `compute ~79-93 ms`보다 `observables ~420 ms`가 더 비쌌다.
- 저장소의 기존 공통 개선안 `figureless-observables-paths`가 바로 이 문제를 겨냥한다 (`docs/checklist.json:28-31`).
- 배치 sweep, 자동 보고서, 테스트에서 metric/table만 필요할 때 figure build를 건너뛰면 원래 기능을 해치지 않고 큰 속도 향상을 기대할 수 있다.

### 2. biphoton solver / topology 재사용

- `compute()`는 biphoton 호출마다 `GenericFWMSolver(topology)`를 새로 만든다 (`gabes/schemes/fwm.py:1860-1865`).
- solver 자체가 아주 무거운 객체는 아니지만, 같은 topology로 반복 sweep할 때는 재사용 여지가 있다.
- 효과는 1번보다 작겠지만, batch scan에서는 깔끔한 미세 최적화가 될 수 있다.

### 3. 2D phase-matching map 계산의 분리 강화

- `phase_detail = Fine`이면 2D angle map을 즉시 계산해서 raw에 넣는다 (`gabes/schemes/fwm.py:943-952`).
- 이미 opt-in이긴 하지만, 큰 sweep에서는 이것도 별도 extra view나 lazy path로 더 강하게 분리할 여지가 있다.
- 이는 기존 checklist의 `fwm-batch-cache-and-map-separation`와도 같은 방향이다 (`docs/checklist.json:70-73`).

## 종합 판단

- 오늘 기준 `FWMScheme`의 seeded 모드는 **실제 85Rb D1 warm-vapor FWM 실험을 준비하고 해석하는 연구자에게 꽤 믿고 쓸 수 있는 semi-quantitative reference**다.
- biphoton 모드는 **실험 geometry와 detector 조건까지 포함한 설계 도구**로 매우 유용하며, predictive path는 6월 24일 상태보다 확실히 성숙했다.
- 다만 두 모드 모두 absolute metrology tool이라기보다는, “어느 knob가 어떤 방향으로 실험을 움직이는가”를 읽는 compact but physically honest reference에 가깝다.
- 지금 가장 좋은 다음 단계는 기존 checklist에 이미 적힌 두 항목, 즉 **per-source biphoton calibration**과 **figureless observables path**다. 전자는 물리 신뢰도를 거의 공짜로 올리고, 후자는 기능 손상 없이 체감 속도를 가장 크게 줄일 가능성이 높다.
