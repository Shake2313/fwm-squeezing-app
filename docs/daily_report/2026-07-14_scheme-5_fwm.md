## 요약 판단

`FWMScheme`은 하나의 이름 아래 서로 성격이 다른 두 도구를 제공한다.

- **Seeded FWM / squeezing**은 실제 `85Rb D1` double-Lambda 실험의 detuning, 온도, gain, 손실, phase mismatch 민감도를 보는 **유용한 semi-quantitative 실험 설계 도구**다. 실제 hyperfine strength, branching, 순수 `85Rb` 밀도, 충돌 dephasing, Doppler 평균, Maxwell-Bloch 전파가 들어 있어 단순 toy model은 아니다.
- **Biphoton**은 vector phase matching, velocity-class 간섭, detector/background, timing jitter, filter bandwidth를 한곳에서 보는 **유용한 정렬·검출계 설계 도구**다. 그러나 absolute pair rate는 독립 예측값이 아니며, 현재는 문서가 주장하는 OD 스케일링과 코드가 일치하지 않는다.
- 따라서 현 코드는 **trend, operating-point 탐색, sanity check에는 좋지만 absolute source metrology의 최종 레퍼런스로 인용해서는 안 된다.** 특히 pair rate, Ultra squeezing 절대값, Cs absolute width에는 보정·문서 정합성 검증이 더 필요하다.

## 현재 구현된 실제 물리

### 1. Seeded `85Rb D1` FWM / squeezing

1. 네 준위 `F=2,3 -> F'=2,3` double-Lambda를 쓰며 각 drive와 polarization readout에 `3 C_F^2`가 들어간다 (`gabes/schemes/fwm.py:26-40`, `1048-1062`). 자발붕괴 branching도 같은 hyperfine 자료로 검증된다 (`tests/test_regression.py:75-93`).
2. 표준 `(-)` Raman branch의 probe/conjugate mode pair를 별도로 풀며, 두 branch susceptibility를 잘못 합산하는 사용을 명시적으로 거부한다 (`gabes/schemes/fwm.py:1414-1430`, `tests/test_regression.py:52-62`). 이는 실험 모드 정의에 맞는 중요한 보호장치다.
3. 각 two-photon detuning과 velocity class에서 3-mode Floquet steady state를 풀고 Doppler 평균한 뒤 2x2 Maxwell-Bloch 전달행렬로 gain을 계산한다 (`gabes/schemes/fwm.py:1113-1188`, `1436-1587`; `gabes/observables.py:17-84`).
4. macroscopic coupling에는 `p_F/[2(2I+1)]`가 들어가고, `line_strength=0.74`는 남은 m-sublevel/geometry 효과를 흡수하는 residual이다 (`gabes/schemes/fwm.py:43-61`, `1461-1468`, `1814-1826`). 이 구분은 실험가에게 비교적 정직하다.
5. 온도에 따라 pure-`85Rb` CRC 밀도뿐 아니라 optical self-broadening과 ground Raman collision dephasing도 다시 계산한다 (`gabes/schemes/fwm.py:1390-1411`, `1477-1485`). 따라서 과거의 단조로운 고온 runaway보다 실제 실험의 내부 온도 최적점을 더 잘 표현한다.
6. Fast부터 진공 phase mismatch, Balanced에서 refractive correction과 16-segment profile, Ultra에서 64-segment overlap/depletion과 arm-resolved loss 및 pump-scatter proxy를 쓴다 (`gabes/schemes/fwm.py:1509-1638`, `1705-1722`). 24-level Zeeman 구조는 CG 합 검사용 진단일 뿐 main Floquet solve가 아니다 (`gabes/schemes/fwm.py:1556-1564`, `1573-1575`).

실험적 장점은 분명하다. 실제 pump/seed power와 waist가 Rabi 주파수로 연결되고, TPD는 cached curve를 이동하는 실험적 knob로 분리되어 있다 (`gabes/schemes/fwm.py:1471-1475`, `1787-1810`). 한편 Manley-Rowe saturation은 에너지 위반을 막는 합리적 guardrail이지만 full nonlinear propagation을 대신하지는 않는다 (`gabes/observables.py:87-115`). Ultra noise도 full quantum-Langevin source term이 아니라 beamsplitter loss와 현상론적 pump-scatter 항이다.

### 2. Generic SFWM / biphoton

1. `87Rb 780/1529 nm telecom cascade`, 두 `133Cs` BTW channel, generic diamond template를 제공한다 (`gabes/schemes/fwm.py:444-590`). Diamond는 검증된 paper preset이 아니라는 표시가 있다 (`gabes/schemes/fwm.py:583-589`).
2. Predictive 경로는 `Omega_c^2` Autler-Townes 항을 보존한 two-photon denominator, Maxwell velocity-class coherent sum, natural-linewidth decay를 사용한다 (`gabes/schemes/fwm.py:710-792`).
3. 좁은 velocity-space source resonance를 먼저 측정하고 FWHM이 3% 이내로 수렴할 때까지 grid를 절반씩 줄인다 (`gabes/schemes/fwm.py:794-873`). 과거 보고서의 velocity aliasing 문제를 실질적으로 해결한 좋은 수치물리 개선이다.
4. longitudinal reference offset과 absolute transverse mismatch를 합친 `sinc^2(|Delta k|L/2)` weight를 쓰고, Fine에서 121x121 angle map을 만든다 (`gabes/schemes/fwm.py:294-323`, `984-995`). 동일 각도나 잘못된 collection side가 강하게 억제되는지 테스트한다 (`tests/test_fwm_generic.py:65-120`).
5. detector efficiency, dark/background, coincidence window, timing jitter, filter bandwidth를 pair rate와 `g2_SI(tau)`, CAR, heralding으로 변환한다 (`gabes/observables.py:174-245`). 이 부분은 실제 검출계 trade-off를 비교하는 데 유용하다.

## 오늘의 직접 수치 확인

### Seeded 기준점

현재 앱 기본값은 `Delta=0.9 GHz`, `delta=-8 MHz`, `121 C`, pump `600 mW`, seed `8 uW`, loss `5.5%`, residual `0.74`, `Fast (~4 s)`이다 (`gabes/schemes/fwm.py:1926-1930`).

| 경로 | 기준점 `G_s` | 기준점 squeezing | scan 최대/최소 | 로컬 시간 |
|---|---:|---:|---:|---:|
| Fast | 5.586 | -6.836 dB | max `G_s=6.419`, min `-7.025 dB` | warm `0.29-0.30 s` |
| Ultra | 13.829 | -7.082 dB | max `G_s=14.813`, min `-7.178 dB` | `4.30 s` |

Ultra gain은 `G_s≈14` anchor와 잘 맞지만, 현재 실행의 squeezing은 문서와 UI help가 말하는 `-7.8 dB` 재현과 약 `0.72 dB` 다르다 (`gabes/schemes/fwm.py:1821-1825`; `docs/FWM_physics.tex:846-853`, `1558-1562`). 숫자 주장을 유지하려면 현 HEAD에서 해당 검증을 테스트로 고정하거나 문구를 `gain anchor, squeezing approximate`로 낮춰야 한다.

### Predictive biphoton 기본값

| topology/channel | FWHM | source bandwidth | raw pair rate | filter 후 pair rate | velocity 수렴 |
|---|---:|---:|---:|---:|---|
| `87Rb` telecom | 0.497 ns | 914.8 MHz | 380 cps | 124.6 cps | true, `dv=0.541 m/s` |
| `Cs` 852-917 | 0.582 ns | 665.3 MHz | 240 cps | 108.2 cps | true |
| `Cs` 852-795 | 2.656 ns | 83.2 MHz | 240 cps | 240 cps | true |

Telecom default width는 repository reference `0.56(4) ns`에 가깝다. 다만 이는 동일 anchor 주변의 한 기본점이며 독립 검증은 아니다. Cs width ratio는 `2.656/0.582 = 4.56`으로, 코드 validation table의 `about 3` target에서 약 52% 벗어난다 (`gabes/schemes/fwm.py:2491-2497`). 따라서 `Cs channels within ~30%`라는 앱 설명은 현재 기본 출력만으로 뒷받침되지 않는다 (`gabes/schemes/fwm.py:2031-2034`).

### 가장 중요한 물리 정합성 문제: pair-rate scaling

문서와 UI는 nonlinear amplitude가 `OD`에 비례하여 rate가 `OD^2`로 스케일하고, pair rate가 pump power·OD·phase matching의 물리 스케일을 따른다고 설명한다 (`docs/FWM_physics.tex:1361-1385`; `gabes/schemes/fwm.py:2291-2296`). 그러나 실제 scalar rate 식은

`R_ref * P_pump * sqrt(coupling_scale) * phase_match * sqrt(N/N_ref)`

이며 cell length나 `od_estimate`를 사용하지 않는다 (`gabes/schemes/fwm.py:923-933`; 식은 `docs/FWM_physics.tex:1455-1472`에도 그대로 적혀 있다).

직접 셀 길이만 바꾼 결과:

| cell | OD estimate | raw pair rate |
|---:|---:|---:|
| 6.25 mm | 56 | 380 cps |
| 12.5 mm | 112 | 380 cps |
| 25.0 mm | 224 | 380 cps |

즉 현재 absolute rate readout은 **reference anchor + 선택된 현상론적 scalar law**이지, 표시문이 말하는 OD-scaled predictive rate가 아니다. 실험가가 cell length나 temperature sweep의 절대 발생률을 해석할 때 가장 위험한 부분이다.

## 기존 개선안과 계산 비용 평가

기존 제안은 충분히 존재하므로 “기존 제안 없음”에 해당하지 않는다.

### 이미 반영되었거나 진행 중

- `biphoton-predictive-model`은 `in_progress`다 (`docs/checklist.json:15-20`). velocity auto-refinement와 waveform-derived bandwidth는 이미 가치가 크다. 오늘 predictive compute는 telecom 약 `0.064 s`, Cs 약 `0.028 s`였으므로 이 보강의 runtime 부담은 작다.
- `figureless-observables-paths`는 완료됐다 (`docs/checklist.json:29-33`). 오늘 seeded headless readout은 약 `0.0001 s`, figure readout은 약 `0.56 s`였다. batch/report에서 물리를 전혀 바꾸지 않고 체감 시간을 줄이는 성공적인 최적화다.
- paper data-availability와 Zeeman 범위 문구는 완료 처리되어 있다 (`docs/checklist.json:36-49`). 다만 앱 `info()`의 `Zeeman diagnostic correction` 표현은 실제로 correction을 적용하지 않는 코드와 다시 어긋난다 (`gabes/schemes/fwm.py:2021-2024` 대 `1573-1575`).

### 거의 무부하로 가능한 기존 제안

- `fwm-per-source-biphoton-calibration` (`docs/checklist.json:67-71`): topology별 linewidth/dephasing/coupling scalar를 분리하는 작업이라 solver 차원을 늘리지 않는다. runtime 증가는 사실상 무시할 수 있지만, 각 scalar의 데이터 출처와 uncertainty를 명시해야 한다.
- `fwm-line-strength-factorization` (`docs/checklist.json:74-78`): residual `0.74`를 overlap, polarization purity, effective participating Zeeman fraction으로 나누어도 계산은 scalar 곱뿐이다. 물리를 잃지 않고 실험 언어를 개선하지만, 서로 퇴화한 factor를 데이터 없이 각각 “측정값”처럼 보이게 해서는 안 된다.
- `fwm-batch-cache-and-map-separation` (`docs/checklist.json:95-99`): topology object는 이미 LRU cache된다 (`gabes/schemes/fwm.py:597-626`, `1040-1042`). velocity grid, scan axis, Fine map을 더 재사용하거나 lazy화하는 것은 동작 보존 최적화다.

### 무부하 개선이 아닌 기존 제안

- Full quantum-Langevin noise (`docs/checklist.json:116-120`)는 source-noise propagation state와 loss coupling을 새로 정의해야 하므로 계산·검증 부담이 크다.
- Full 24-level Zeeman Floquet (`docs/checklist.json:123-127`)는 density-matrix 좌표가 `M=16`에서 `M=576`으로 커진다. dense solve의 단순 cubic 비율만 보아도 매우 무거우며, 지금의 interactive default에 넣을 작업이 아니다. 진단 manifold와 별도 opt-in 연구 경로로 유지하는 판단이 타당하다.

## 이번 근거로 제안하는 저비용 개선

### P1 후보 — computed waveform을 이용한 relative rate scaling

Predictive 경로는 이미 `psi_tau`, OD, drive, detuning, phase matching을 계산한다. 따라서 별도 OBE/Floquet solve 없이 `integral |psi(tau)|^2 dt`를 reference 조건의 동일 norm으로 나누고, absolute anchor `R_ref`만 유지하면 된다. 이렇게 하면 reference point의 38000 cps/mW는 보존하면서 cell length, density, detuning, coupling의 상대 변화가 실제 계산한 source amplitude를 따른다. 추가 비용은 배열 norm과 topology별 reference norm cache 정도로 무시할 수 있다. 현재의 `sqrt(N)` 식을 계속 쓸 경우에는 적어도 UI를 “empirical sqrt-density scaling, no cell-length scaling”으로 정확히 낮춰야 한다.

### P2 후보 — 수치 주장과 실행 예의 단일 기준화

- Ultra `-7.8 dB` 재현 문구를 현 HEAD 검증값과 맞춘다.
- README의 residual 기본 `1.0`/“leave at 1.0”을 앱 기본 `0.74`와 구분한다 (`README.md:155-165` 대 `gabes/schemes/fwm.py:1814-1826`).
- README의 `chi table is T-independent`를 충돌 dissipator가 들어간 현재 `per-T table`로 고친다 (`README.md:200-206`; `gabes/schemes/fwm.py:1390-1404`).
- `docs/FWM_physics.tex`의 예전 `High fidelity` 명칭을 현재 `Fast / Balanced / Ultra`로 갱신한다 (`docs/FWM_physics.tex:78-81`, `759-779`; `gabes/schemes/fwm.py:1707-1733`).
- 앱 `info()`의 `self-consistent phase refinement`와 `Zeeman correction`을 실제 `single-pass refractive update`, `diagnostic only`로 바꾼다 (`gabes/schemes/fwm.py:1324-1345`, `1556-1575`, `2003-2024`).

이들은 runtime 비용이 0이며, 실험 레퍼런스로서의 신뢰를 즉시 높인다.

### P2 후보 — CLI/example density 정합성

`fwm_obe.py`는 화면에 `atoms.rb85_density(T)`를 출력하지만 실제 `compute_spectrum()`은 pure-`85Rb` `hyperfine.number_density(T)`를 쓴다 (`fwm_obe.py:67-85`; `gabes/schemes/fwm.py:1477-1485`). `121 C`에서 전자는 `1.272e19 /m^3`, 후자는 `2.150e19 /m^3`로 표시값이 실제 계산값의 0.592배다. 출력 한 줄을 실제 backend density로 통일하면 계산 비용 없이 예제의 재현성이 좋아진다.

### 테스트 보강

- 현재 baseline 회귀는 `PHASE_LEGACY` direct path를 고정하지만 앱 기본 Fast/Ultra의 absolute anchor는 고정하지 않는다 (`tests/test_regression.py:31-49`, `tests/test_regression.py:96-101`). 수치 재현을 문서에 쓸 경우 기준점 `G_s`, `S_dB`를 별도 tolerance test로 고정해야 한다.
- cell length 변화에 pair rate가 어떻게 반응해야 하는지 test가 없다. reference point 보존과 low-OD relative `OD^2` 경향을 함께 검증하는 테스트가 필요하다.
- Cs test는 width ordering만 확인한다 (`tests/test_fwm_generic.py:200-213`). `about 3` 또는 `within 30%`를 주장하려면 ratio tolerance를 실제 test로 고정하거나 문구를 qualitative ordering으로 낮춰야 한다.

## 순수 코드 최적화

### 이미 들어온 큰 개선

현재 작업 트리의 FWM kernel은 전체 `(delta, Delta_eff)` grid를 Numba에서 fuse하고, `+/-` sideband conjugate symmetry로 두 번째 matrix factorization을 제거한다 (`gabes/kernels.py:166-273`; `gabes/core.py:143-171`). 양 branch에서 NumPy reference와 상대 오차 `<1e-10`을 확인하는 테스트가 있다 (`tests/test_kernels.py:28-59`).

오늘 `analysis/bench_numba_fwm.py` 실행 결과:

| workload | NumPy | Numba | speedup | max `|Delta S|` |
|---|---:|---:|---:|---:|
| scan-point | 1.36 s | 0.12 s | 11.6x | `8.63e-13 dB` |
| app-fast benchmark | 1.96 s | 0.14 s | 13.7x | `5.63e-14 dB` |
| app-fine benchmark | 14.96 s | 1.03 s | 14.5x | `3.91e-14 dB` |

이는 기능을 바꾸지 않고 큰 속도 향상을 얻은 좋은 최적화다. 다만 benchmark 파일의 `Fast=121 points`, `Fine` 설명은 현재 앱의 `Fast=181`, `Balanced=301` 명칭과 달라 갱신이 필요하다 (`analysis/bench_numba_fwm.py:7-31`; `gabes/schemes/fwm.py:1707-1722`).

### 추가 동작 보존 후보

1. `compute_spectrum()`은 같은 Doppler interpolation index/fraction을 네 susceptibility table마다 다시 계산한다 (`gabes/schemes/fwm.py:1502-1507`; `gabes/doppler.py:35-68`). `interpolation_weights()`를 한 번 계산하여 네 번 `apply_doppler_average()`에 재사용하거나 네 table을 stack해 한 번 contract하면 allocation과 index 계산을 줄일 수 있다.
2. Fine 121x121 map은 이미 opt-in이지만 raw compute 안에서 즉시 만든다 (`gabes/schemes/fwm.py:984-995`). figure나 extra view를 실제 열 때 lazy 계산하면 대규모 batch의 accidental Fine 비용을 없앨 수 있다.
3. Cs reference table은 선택하지 않은 다른 channel을 observables 단계에서 다시 compute한다 (`gabes/schemes/fwm.py:2471-2488`). topology/physics parameter key로 cross-channel validation 결과를 cache하면 headless batch의 중복 계산을 없앨 수 있다.
4. repeated sweep에서는 cached topology뿐 아니라 `(T, resolution)` velocity grid와 focused scan axis를 읽기 전용 cache로 재사용할 수 있다. 이는 기존 `fwm-batch-cache-and-map-separation`과 같은 방향이며 물리 결과를 바꾸지 않는다.

## 검증 결과

- 첫 targeted test 실행은 로컬 Tcl/Tk 설치 문제로 figure test 1개가 실패하고 45개가 통과했다. traceback은 `matplotlib`의 Tk window 생성에서 발생했으며 FWM 계산 실패가 아니었다.
- 비대화형 backend로 저장소 정책의 전체 명령을 다시 실행했다: `MPLBACKEND=Agg python -m pytest -q` -> **118 passed in 26.36 s**.
- seeded reference point, Ultra point, three predictive biphoton channels, cell-length/temperature rate sweep, Numba/NumPy benchmark를 별도로 실행했다.

## 최종 결론

Seeded FWM은 현재도 GABES에서 가장 물리적으로 풍부한 스킴 중 하나다. 실제 `85Rb D1` hyperfine 구조, branch convention, 충돌 dephasing, Doppler 평균, phase mismatch와 loss-aware Ultra readout은 실험 동작점·민감도 탐색에 충분한 가치가 있다. 다만 residual calibration과 현상론적 saturation/noise 때문에 absolute metrology 도구로는 보수적으로 써야 한다.

Biphoton은 geometry와 detector trade-off reference로는 매우 유용하고 velocity-grid 수렴 개선도 성공적이다. 그러나 absolute pair rate는 현재 설명과 달리 OD 또는 cell length를 따르지 않는다. **가장 우선할 저비용 개선은 reference anchor를 유지하되 이미 계산한 waveform norm으로 relative rate를 스케일하는 것**, 그리고 Ultra/CS/default/fidelity 문서를 현 HEAD에 맞추는 것이다. Full Langevin이나 full Zeeman으로 바로 확장하기 전에 이 저비용 정합성 문제를 해결하는 편이 실험 레퍼런스 신뢰도 대비 비용이 훨씬 좋다.
